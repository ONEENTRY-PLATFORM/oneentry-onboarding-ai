# OneEntry invariants — hard rules of the blueprint

> **⚠ Universality note.** Examples below frequently use fashion-shop terms (clothing / shoes / bags / women / men) because that is the reference test project. The **rules themselves are universal**: substitute the vocabulary of YOUR project type when applying them — e-commerce shop (`product/sku/brand/category`), restaurant (`menu-item/dish/cuisine/section`), beauty salon (`service/master/treatment/duration`), hotel (`room/suite/amenity`), EdTech (`course/lesson/level`), corporate site (`page/department/team`), personal cabinet (`section/setting/subscription`), SaaS (`plan/feature/seat`). The blueprint pipeline treats all of them the same way.

> **This file is a self-contained mirror of OneEntry Platform rules.** Agents use only this file.

These invariants are mandatory for **all** projects. If an application structurally violates them (for example, has two separate forms for login and signup), the mapper's job is to **bring it into canonical form**, not pass it through "as is".

## 1. Authentication form = one (login = signup)

**Rule:** `forms` contains **exactly one** record with `type: 'sing_in_up'` (typo carried over from the source enum, but this is the actual value), which serves both login and signup.

**Login vs signup distinction** — at the level of attributes in this form's attribute_set schema:
- an `email` attribute (or `phone`, `login`) with `isLogin: true`
- a `password` attribute with `isPassword: true`
- a `sign_up` attribute (boolean) with `isSignUp: true`

If the frontend sends `signUp=true` → the backend interprets it as registration, otherwise — login.

Never:
- ❌ two separate forms `'@form.login'` and `'@form.signup'`
- ❌ a form with `isLogin` but no `isPassword`
- ❌ a form with `isSignUp: true` on multiple attributes

## 2. Standard user_groups (minimum)

Every project logically has two STOREFRONT user groups:

| identifier | id in DB | localized title (en_US) | Purpose |
|---|---|---|---|
| `guest` | **1 (preseeded)** | Guest | Unauthenticated visitors — **already in the DB, do not generate in the blueprint** |
| `user` | auto | User | Registered storefront users — generated by the blueprint |

⚠ **DO NOT create an `admin` user_group via blueprint.** CMS administrators are managed by the dedicated `admins` module (created via `npm run seed:admins` in the CMS), NOT through `user_groups`. The `user_groups` table is a STOREFRONT concept — adding an `admin` row there creates a useless storefront group that is confused with cms-admins. Verified against the CMS `set-default-user-group` seed (seeds ONLY `guest`).

⚠ `guest` is preseeded in any fresh OneEntry Platform DB (see `rules/preseeded-entities.md`). The mapper must NOT include it in `user_groups[]`. For FK references to guest, use the numeric `user_group_id: 1`.

Optionally (only if explicitly present in the code):
- `manager`, `vip`, `wholesale`, `b2b` — extended storefront groups.

## 3. Standard product_statuses (minimum)

| identifier | localized title (en_US) |
|---|---|
| `active` | Active |
| `draft` | Draft |
| `archived` | Archived |

These three are always set up, even if the project has no notion of product status. `is_default: true` — on `active`.

## 4. Standard order_statuses (MANDATORY when orders subsystem is included)

If the project has ANY of: a cart page, a checkout flow, an `OrderStatus`/`orderStatus` enum, or order REST signals (see `rules/standard-entities.md` → "orders_storage + order_statuses" for the full trigger list) — emit these **exactly 4 statuses** verbatim:

| identifier | localized title (en_US) | is_default |
|---|---|---|
| `new` | New | true |
| `processing` | Processing | false |
| `done` | Done | false |
| `cancelled` | Cancelled | false |

Why these 4: the deterministic `post-mapper-fixer.generate_payment_status_maps()` builds the admin "Payment Status Settings" mapping by matching these identifiers against the 5 `PaymentStatusMapDto` keys (`waiting → new`, `partial → processing`, `completed → done`, `canceled/expired → cancelled`). Any deviation breaks the auto-mapping → admin will see an empty "Payment Status Settings" page.

`orders_storage.general_type_id` is **STABLE = 21** (`'order'`, see `rules/dynamic-ids.md`). Always emit `general_type_marker: 'order'` next to it for build-time verification.

⚠ `order_statuses.storage_id` — NOT NULL → `orders_storage` row must come first in `mapped.yaml`. The previous wording "if you have no idea what to put in `general_type_id` — skip the whole thing" is **OBSOLETE**: the id is known and stable.

## 5. System attribute flags — exactly 1 per attribute_set

| Flag | Where | Count in one attribute_set |
|---|---|---|
| `isPrice` | A `real` attribute on a product | exactly 1 |
| `isSku` | A `string` attribute on a product | exactly 1 |
| `isCurrency` | A `string` attribute on a product (if present) | ≤ 1 |
| `isProductPreview` | An `image` attribute on a product | exactly 1 |
| `isPassword` | A `string` attribute on the login=signup form | exactly 1 |
| `isLogin` | A `string` attribute on the login=signup form | exactly 1 |
| `isSignUp` | A `radioButton` attribute on the login=signup form | exactly 1 |
| `isTaxRate` | A `real` attribute (if present) | ≤ 1 |

The builder MUST verify this and fail if the rule is violated (the validator performs the same check).

## 6. Identifiers — ASCII snake_case

- Only latin characters + digits + `_`.
- No whitespace, no Cyrillic, no special characters.
- Predictable names: `title`, `description`, `price`, `sku`, `color`, `size`, `images`, `preview`, `email`, `password`, `sign_up`, `weight`.
- Unique within an attribute_set schema.

## 7. Localization — every language from `inspector.yaml.detected_languages`

In `localizeInfos` / `localize_infos` (for forms, attributes, pages, statuses, templates, user_groups, products, blocks) — **iterate over every language in `inspector.yaml.detected_languages`** (see `agents/code-inspector.md` Step 1.2). This is an array of 1..~10 OneEntry keys (`en_US`, `de_DE`, `fr_FR`, `it_IT`, ...).

**Rules:**

- **At least 1 language, at most ~10.** If the array is empty — this is itself an inspector bug; the mapper uses fallback `['en_US']` and emits a warning.
- **All languages share THE SAME set of keys** in `localize_infos`. If for a page entity this is `{title, plainContent, htmlContent, menuTitle}` — every language must contain **all four** keys.
- **The default language** (`inspector.yaml.language`) must be in `detected_languages`. If the project has only one language — the array contains one element.
- **Value population:** for the default language — the real text from the code. For other languages — either the same value (if no translation is available; the admin will translate later) or the real translation from an i18n dictionary if the inspector found one.

### Default-language detection algorithm (used only if detected_languages is empty)

1. Read 5-10 visible headings from the source (titles, button labels, page headings).
2. Count the share of non-ASCII characters in them and detect the dominant locale.
3. If >50% non-ASCII — pick the dominant locale (e.g. `de_DE`), otherwise — `en_US`.

### Examples

**Single-language (legacy / small project):**

```json
{
  "localize_infos": {
    "en_US": { "title": "Home page", "plainContent": "", "htmlContent": "<p></p>", "menuTitle": "Home" }
  }
}
```

**Bilingual (en_US + de_DE):**

```json
{
  "localize_infos": {
    "en_US": { "title": "Home",      "plainContent": "", "htmlContent": "<p></p>", "menuTitle": "Home" },
    "de_DE": { "title": "Startseite","plainContent": "", "htmlContent": "<p></p>", "menuTitle": "Startseite" }
  }
}
```

**Multi-locale (EU shop, 5 languages):**

```json
{
  "localize_infos": {
    "en_US": { "title": "Home",    "plainContent": "", "htmlContent": "<p></p>", "menuTitle": "Home" },
    "de_DE": { "title": "Startseite","plainContent": "", "htmlContent": "<p></p>", "menuTitle": "Startseite" },
    "fr_FR": { "title": "Accueil",   "plainContent": "", "htmlContent": "<p></p>", "menuTitle": "Accueil" },
    "it_IT": { "title": "Home",      "plainContent": "", "htmlContent": "<p></p>", "menuTitle": "Home" },
    "es_ES": { "title": "Inicio",    "plainContent": "", "htmlContent": "<p></p>", "menuTitle": "Inicio" }
  }
}
```

The keys of the object are language codes from `detected_languages`. Do not use arbitrary codes (`en`, `de` without country) — only OneEntry format.

## 8.5. Catalog hub pages — universal `general_type_id` invariant

Any page that has a `catalog_page` (general_type_id=4) descendant — at ANY depth — **must itself be `catalog_page`**, even if it has no products directly. Mapper must NOT classify category hubs as `common_page` (gtid=17).

**Why:** cms admin catalog list (`developer-pages.controller.ts::getTableData2`) walks only ONE level deep when testing for catalog descendants:

```sql
WHERE p.parent_id = :parentId
  AND (types.type='catalog_page' OR EXISTS (
    SELECT 1 FROM pages nested_p
    INNER JOIN general_types nested_t ON nested_p.general_type_id = nested_t.id
    WHERE nested_p.parent_id = p.id AND nested_t.type = 'catalog_page'
  ))
```

An intermediate hub `common_page` whose own direct children are themselves common-page hubs gets HIDDEN by the admin tree — the user clicks into Catalog and sees an empty branch (their catalog leaves are 2 levels down, not directly nested).

**Universal across verticals (always promote these hub chains to gtid=4):**

| Project type | Example hub chain |
|---|---|
| E-commerce shop | `catalog → women → women-clothing` |
| Restaurant | `menu → desserts → ice-cream-vanilla` |
| Hotel | `rooms → suites → presidential` |
| Education | `faculty → cs → algorithms` |
| Real estate | `properties → residential → houses` |
| Marketplace | `services → consulting → tax` |

`post-mapper-fixer.py` Fix 3.5 implements this universally via recursive descendant walk over `pages[]`.

**Counter-examples — always stay common_page / their own type:**

- `root` (orchestrates the whole tree, no parent)
- Utility: `cart`, `checkout`, `favorites`, `account`, `wishlist`, `profile`
- Error: `404`, `500`, `503`, `offline`, `error`, `not-found`
- Auth: `login`, `signup`, `forgot-password`

## 9. Root page — exactly one with parent_id=null

```json
{ "id": "@page.root", "identifier": "root", "parent_id": null, "page_url": "", ... }
```

All other pages link to it via `parent_id: '@page.root'` or via a chain (e.g. `@page.cart` → `@page.root`).

## 10. attribute_set schema — at least 1 item (with one exception)

Every attribute_set's `schema` must contain at least one attribute.

Exception: technical attribute_sets without custom fields (for admin, for a technical group). They may have `schema: {}`.

## 11. Never set position_id manually

The loader will create a `positions` row itself via `auto_positions=true` (default). The builder **does not** include a `position_id` field on blueprint rows.

Exception: if you really want to hardcode a position, you would have to go through polymorphic `positions` directly — but `positions` is **outside the whitelist**, so you cannot anyway. The rule is simple: **never position_id**.

## 14. Idempotence

Re-running the blueprint collector on the same source produces an **identical** JSON. No:
- UUIDs
- timestamps in the data
- Math.random()

All tokens and identifiers — deterministic hashes/slugs from the source fields (filename, sku, title).

## 15. Blocks — usage rules

**What a block is in OneEntry:**
a block (`blocks` table) is a **reusable content section** attached to a page or product (for example, hero-slider, related-products, faq, banner, collection-of-products). A block is NOT a template (template = the page's overall layout). A block is NOT just a UI component.

### 15.1 When something in the code is a block

Signs of a content block (created as a `blocks` row):

1. A JSX/HTML section **embedded in several pages the same way** (hero on home, hero on category, hero on sale).
2. A "similar products / recently viewed / reviews" section on a product page → a block (on the product page via `block_products_mn`).
3. CMS-managed sections on static pages: "about us", "testimonials", "hero", "featured collection" → a block.
4. Sliders/carousels/banners on the home and category pages → a block.
5. "New arrivals", "Sale", "Top views" blocks — typical blocks.

### 15.2 What is NOT a block

- **Header / Footer / Navigation / Sidebar** — part of the `template` level (the app's shared layout), not a content block. Do not create `blocks` rows for them. If a "templates" subsystem appears in the project — those belong there.
- **Buttons, form fields, product list items** — UI primitives, not blocks.
- **Whole pages** (`HomePage`, `CartPage`) — these are `pages`, not blocks.

### 15.3 One block — many pages / many products

If the same block (one semantic unit, the same schema) is used on **N** pages:
- Create **one** record in `blocks` (with one `attribute_set_id`).
- Create **N** records in `block_pages_mn` (one per page), all pointing to the same `block_id`.

❌ Anti-pattern: creating `blocks: hero_home`, `blocks: hero_category`, `blocks: hero_sale` when they share the same schema. There should be one `blocks: hero` + three `block_pages_mn` records.

### 15.4 Block attribute_set — type_id=2 (forBlocks) mandatory

Every block references its `attribute_set` via `attribute_set_id`. That attribute_set MUST have `type_id: 2` (`forBlocks`).

❌ Do not use `forPages` or `forProducts` attribute_set for a block — that violates type semantics.

If several blocks have **identical schemas** (same attributes with the same types) — they share **one** `attribute_set` (type_id=2). Otherwise each block has its own attribute_set.

### 15.5 Do not confuse a block with a template

| Concept | Where |
|---|---|
| **template** | The whole page layout (`general_type_id` 2 = catalog, 3 = product, 4 = page) — defines "how the page is drawn" |
| **block** | A reusable section inside a page (hero-slider, faq, collection) — defines "what content is in the section" |

In a Next.js codebase:
- File `pages/HomePage.tsx` or `app/page.tsx` → this is a **page** (or its template).
- Components imported into HomePage like `<HeroSlider/>`, `<NewArrivals/>`, `<PromoBlock/>` → these are **blocks**.
- Components `<Header/>`, `<Footer/>` — **layout, not blocks** (part of the template, not added to `blocks`).

### 15.6 A block without links — anti-pattern

A `blocks` record with no link in `block_pages_mn` / `block_products_mn` / `product_blocks_mn` is an orphan block, useless from a CMS runtime perspective. The validator should flag it as a warning. The builder should either attach every block to at least one page, or not create it.

## 16. Unification rules — deduplication without breaking structure

Goal: remove accidental duplicates (identical forms / attribute_sets / blocks that arose from symmetry in the source code), but **without breaking the project's semantics**.

### 16.1 Mandatory merge rules

Always applied (the mapper has no right not to merge):

| Candidate | Merge condition | Result |
|---|---|---|
| login + signup forms | by definition (see invariant #1) | one `signin` form (`type: 'sing_in_up'`) |
| user_groups `user` | standard name | reuse, do not multiply (guest is preseeded — skip entirely; admin is NOT a user_group, it's in the `admins` module) |
| product_statuses active/draft/archived | standard names | reuse |
| attribute_sets `forUserGroups` | empty schema | one shared across all groups |
| attribute_sets `forAdmins` | empty schema | one technical one for admin |

### 16.2 Conditional merge rules (by signature)

Applied when two candidates share a **signature** = `(type, set_of_attribute_identifiers, set_of_attribute_types)`.

**Algorithm:**
1. For each candidate compute `signature_hash` — a stable serialization (sorted keys) of the attribute set with types and system flags.
2. Group candidates by `signature_hash`.
3. If a group has >1 — merge into one, picking the **shortest or most semantic** identifier.
4. Record the merge decision in `<project>.log.md` with the list of merged identifiers and the rationale (identical schema_hash).

**Applies to:**
- `attributes_sets` (identical `schema` → one reusable attribute_set).
- `forms` (same field set + same `type` enum → one form).
- `blocks` (identical attribute_set schema + same identifier pattern → one block).
- `pages` (same attribute_set + same URL semantics — e.g. `/login` and `/signup` share auth semantics → one `account` or `signin` page).

### 16.3 When NOT to merge (semantic protection)

Unification **must not break the project's structure**. Do not merge when:

1. **Different semantics with identical schema.** Contact form {name, email, message} and feedback form {name, email, message} — even if the schemas match, if they are distinct endpoints in the code (`/contact`, `/feedback`) with different processing → two forms.
2. **Different `type` enum values.** Form `'sing_in_up'` and form `'order'` cannot be merged.
3. **Different `processing_type`.** A form with `'db'` and a form with `'email'` are not merged.
4. **Standard entities.** `user` user_group — never merge with other roles. `guest` — preseeded, not generated at all (see `rules/preseeded-entities.md`). `admin` is NOT a user_group at all — cms admins live in the `admins` module (see §2 and §16.1).
5. **Different attribute_set type_id.** `forProducts` (5) and `forBlocks` (2) — even with identical schemas, do not merge; different semantic levels.
6. **Different block "kinds".** A hero-slider and a review-block with identical `{title, items}` schemas may accidentally match but are semantically different. In that case — keep separate, add a warning "possible unification candidate".

### 16.4 Mapper algorithm

For each candidate group (attribute_sets, forms, blocks):
1. Collect all candidate entities from the inspector + the standards.
2. Compute `signature(entity) = (type, sorted(attribute_identifiers), sorted(attribute_types))`.
3. Group by signature.
4. For groups with size > 1:
   - If they match rule 16.1 → merge unconditionally.
   - If they match rule 16.2 (condition) and NOT rule 16.3 (exception) → merge with warning "merged because of identical signature".
   - Otherwise → keep separate with warning "similar signature, not merged due to <reason>".
5. Record in log.md a `## Unification decisions` section with the merged identifiers and the reason.

### 16.5 Standard entities — reuse, do not multiply

- If the code has several places with email+password — that is the same signin form (see invariant #1).
- If the code has several places with {name, address, phone} — that is potentially the same address attribute_set (though usually address is a user field, not a separate entity).
- If the code has "categories" in several places (list page, filter, navigation) — that is the same `pages` table, do not duplicate.

## 17. Block default attribute_set

Minimal attribute_set for blocks (recommendation):

```yaml
- identifier: 'forBlocks_default'
  type_id: 2          # forBlocks
  schema:
    title:       { type: string, position: 1, identifier: title,       localizeInfos: { <lang>: { title: 'Title' } } }
    description: { type: text,   position: 2, identifier: description, localizeInfos: { <lang>: { title: 'Description' } } }
```

If a block has a more complex schema (slides, items, images) — add fields on top, but reuse `forBlocks_default` if it suffices.

See also §18 (Anti-Hallucination) — even for `forBlocks_default` the localized `title`/`description` of the attributes is taken only from real sources in the target project's code.

## 18. Anti-Hallucination — mandatory source-refs for string values

All string values that end up in `localize_infos.*` (`title`, `menuTitle`, `plainContent`, `htmlContent`, `mdContent`, `description`, `successMessage`, `unsuccessMessage`) and in `attributes_sets.<lang>.<attr>` for attributes of type `string` / `text` / `textWithHeader` **must have a direct source in the target project's code**. It is forbidden to "reconstruct" these values by analogy, from the identifier, or from "common sense".

### ✅ Allowed sources

1. Heading tags in .tsx/.jsx/.vue — `<h1>TEXT</h1>`, `<h2>TEXT</h2>`, `<h3>TEXT</h3>` (`MenCatalogPage.tsx:86 → <h1>CLOTHING</h1>`).
2. A prop passed to a component as an explicit string — `title="TEXT"`, `label="TEXT"`, `placeholder="TEXT"` (`<CategoryHero title="CLOTHING" />`).
3. A string literal in a config file — `categories.ts`, `menu.ts`, `navigation.ts`, `breadcrumbs.ts` (`SUB_CATEGORIES = ['CLOTHING', 'SHOES', ...]`).
4. A breadcrumb / labels / nav config with an explicit value — `{ label: 'Clothing', href: '/men/clothing' }`.
5. Next.js SEO metadata — `export const metadata = { title: 'X' }`, `<title>X</title>`.
6. Real text in an i18n file — `messages/<lang>.json`, `locales/<lang>.json`, `i18n/<lang>.json` (source = `<file>#<jsonpointer>`, e.g. `messages/en.json#/pages/about/title`).
7. Text from `.md` / `.mdx` content (`content/pages/about.md` front matter `title:` or the first `# heading`).
8. A module constant exported as a ready-made string — `export const PAGE_TITLE = 'About Us'`.

### ❌ Forbidden sources (= hallucination)

1. **The file or folder name.** `men-clothing.ts` → `Men's Clothing` — **NO**.
2. **Title Case from an identifier.** `men-clothing` → `Men's Clothing` / `Men Clothing` / `Men-Clothing` — **NO**.
3. **Analogy with another project / another page.** "In the previous project `about` was called `About Us`, I'll set it here too" — **NO**.
4. **"Common sense" / "by context" / "this is how it's usually written".** — **NO**.
5. **Lorem ipsum / placeholder.** `'Title here'`, `'TODO'`, `'Untitled'`, `'New page'`, `'Sample'` — **NO**.
6. **Machine translating the identifier into natural language.** `cart` → `Cart` (translated into another locale) without an explicit source for the text — **NO**.
7. **Derived from the route URL.** `/men/clothing` → `Men Clothing` — **NO**.
8. **Copying a sibling language's value as a "translation".** If `de_DE.title='Warenkorb'` exists, you must not automatically write `en_US.title='Warenkorb'` without an explicit en_US source — **NO**.

### Behaviour when there is no source

1. **inspector** sets `source: 'NOT_FOUND'` (or `'NOT_FOUND_DYNAMIC'` if the value is composed from a template string / variables).
2. **mapper** leaves the corresponding field = `null` and does **not** substitute a derivation from the identifier.
3. **mapper** adds to `mapped.warnings` a string formatted as: `"missing_title: pages.<identifier>.localize_infos.<lang>.title — source NOT_FOUND, left null for the admin"`.
4. **blueprint-builder** writes `null` without inventions.
5. **validator** (S36 in `blueprint-validator.md`) catches suspicious cases where `title` matches the Title Case of the identifier — this is a WARNING, not an ERROR.

### Edge cases

- **Title via CSS uppercase** (`text-transform: uppercase` + `title="Clothing"` → visually `CLOTHING`) — write what is in the code: `'Clothing'`. CSS transformation is not our concern.
- **Title hardcoded into a config object** — the source is valid, record the file path + line.
- **Title built dynamically** (`\`${gender} ${category}\``) — `source: 'NOT_FOUND_DYNAMIC'` + warning.
- **Title in an i18n JSON** — source = path + JSON pointer of the key. If i18n is not detected by grep — fall back to `NOT_FOUND`.
- **Title passed via a prop** (`<h1>{props.title}</h1>`) — climb the stack to find an explicit string; if grep does not find one — `source: 'NOT_FOUND_DYNAMIC'`.

### Justified exceptions (NOT hallucination)

Three narrow categories of derived titles are allowed despite no direct source in code. They are deterministic UX defaults built from project-agnostic tokens, not Title-Case translations of project-specific identifiers.

> ⚠ **Vertical-defaults disclaimer.** The dictionaries cited in exceptions 1, 2, and 3 (`hub_titles`, `composite_catalog`, `block_default_titles`) are populated with **fashion-shop / e-commerce vocabulary** derived from the reference test project. For projects in other verticals (hotel, restaurant, LMS, B2B SaaS, etc.) the dictionaries must be extended in `agents_datasets/scripts/shared/title-derivations.json` with the vertical's actual hub/leaf/block identifiers. The exception **algorithm** is universal — but a `women-shoes` page in a wholly different vertical that happens to share the identifier will still get `"Women's Shoes"` until the dictionary is overridden per-project. Adding entries is non-destructive; the algorithm picks deterministic lookups by identifier.

1. **Hub pages** — atomic navigation nodes. In the fashion-shop reference: `root`/`home`/`women`/`men`/`cart`/`checkout`/`account`/`info`/... — see `entity-mapper.md` Step 7 «Exception to NO HALLUCINATION for hub pages». Title comes from `agents_datasets/scripts/shared/title-derivations.json` → `hub_titles` (e.g. `cart` → "Shopping Cart"). Other verticals extend with their own hubs (hotel: `rooms`, `suites`; restaurant: `menu`, `drinks`; LMS: `courses`, `tutorials`).

2. **Catalog-leaf composite pages** — identifier matches `{head}-{leaf}` where `head` is in `composite_catalog.genders` AND `leaf` is in `composite_catalog.leaves` (both lists live in `title-derivations.json`). Title is composed from `gender_possessive[head] + ' ' + hub_titles[leaf]`. In the fashion-shop reference: `genders ∈ {women,men,kids,unisex}` × `leaves ∈ {clothing,shoes,bags,accessories,sale,new,new-arrivals}`. Example: `women-accessories` → `"Women's Accessories"`. Implementation: `post-mapper-fixer.py::derive_page_title()`. **Does NOT** apply to identifiers where `head` is not in `composite_catalog.genders` (e.g. `outdoor-bags`, `summer-clothing` for fashion; `family-suites`, `seaview-villas` for hotels until vertical override) — those still require a real source in code.

3. **Blocks with null title from source** — neutral UX label from `title-derivations.json` → `block_default_titles` (e.g. `faq` → `"Frequently asked questions"`, `hero_slider` → `"Hero slider"`). The mapper first tries to extract a real `<h1>` / `<h2>` / `SECTION_TITLES.X.title` from the component file (`post-mapper-fixer.py::extract_block_title()`); only when that fails the JSON default kicks in. Defaults must be semantically compatible with the block's `general_type_id` (see validator S47): for example, `catalog_cross_sell` (gtid=30 `cart_complement_block`) must default to `"Complete your look"`, NOT `"You may also like"` (which maps to `similar_products_block` gtid=8). Some defaults like `"Complete your look"` are fashion-flavored — non-fashion verticals can override per project.

Anything outside these three categories falls back to the original rule: leave `null` and emit `missing_title` warning.

See also:
- `agents_datasets/agents/code-inspector.md` §3.6 — how the inspector populates `source`.
- `agents_datasets/agents/entity-mapper.md` Step 7 (localize_infos / NO HALLUCINATION rule + hub/composite exceptions).
- `agents_datasets/agents/blueprint-validator.md` S36 — synthetic-title detector (skips hub + composite leaves).
- `agents_datasets/agents/blueprint-validator.md` S58 — null-title detector for hub/catalog pages (errors).

## 19. Hub pages vs Catalog pages — `general_type_id` discipline

`pages.general_type_id` is the **runtime contract** between admin UI / storefront / loader. Choosing the wrong value either breaks the admin editor (no template attached) or hides facets/products on the storefront.

| general_type_id | name | What it is | Storefront expectation |
|---|---|---|---|
| **3** | `error_page` | 404 / 500 | one record per project, always `parent_id=null` allowed (it's separate from the tree) |
| **4** | `catalog_page` | A listing of products with filters/sorting | Storefront calls `GET /api/content/pages/:id/products` + `GET /api/content/filters/marker/:identifier`. Must have `forProducts` attribute_set associations via `products_pages_mn`. |
| **17** | `common_page` | Generic content page (about, contact, landing, hub-landing) | Storefront renders blocks + page-level localize_infos. No product listing. |

### 19.1 Hub-page anti-pattern

A "hub page" is an intermediate page that has children but no products of its own:

- `/women` (hub) → children `/women/clothing`, `/women/shoes`, `/women/bags` (catalogs).
- `/men` (hub) → children `/men/clothing`, `/men/bags`, ...
- `/discover` (hub) → children `/discover/articles`, `/discover/lookbooks`.

A hub-page has **content** (banners, category cards, hero image) but **no facet-filtered product listing**. Use `general_type_id=17` (common_page) for hubs.

Children of a hub that DO list products use `general_type_id=4` (catalog).

```yaml
pages:
  - id: '@page.women'              # hub
    identifier: women
    general_type_id: 17            # ✅ common_page — content, no facets
    parent_id: '@page.root'
    template_id: '@template.common_page_default'

  - id: '@page.women-clothing'     # child catalog
    identifier: women-clothing
    general_type_id: 4             # ✅ catalog_page — facets + products
    parent_id: '@page.women'
    template_id: '@template.catalog_page_default'
```

### 19.2 Wrong patterns

- ❌ Hub page with `general_type_id=4` but no products attached → admin sees empty catalog, storefront shows "no products". Use 17.
- ❌ Catalog page with `general_type_id=17` → storefront route doesn't load filters, no product list. Use 4.
- ❌ All catalog descendants of a hub sharing the same gtid as the hub. Decide per page based on what the page renders.

### 19.3 Validator S41 already catches half of this

S41 (`general_type_id semantic`) warns when the page identifier suggests a catalog (`-shop`, `-catalog`, `-products`, `-listing`) but gtid=17, and vice versa. Extend S41 to detect hub pages: if a page **has children** AND the identifier matches `^(women|men|kids|sale|brands|categories|shop)$` style hub names → expect gtid=17 unless products are also attached via `products_pages_mn`.

### 19.4 Source of truth

See `agents_datasets/rules/general-types.md` for the full general_type_id enum. See `entity-mapper.md` Step 7 (pages) for the heuristic. The validator implementation lives in `blueprint-validator.md` S41/S58.

## 20. Filters — out-of-whitelist post-import setup

Filters (`filters` / `filter_items_mn` / `filter_custom_items_mn` tables) are **not** in the 24-table whitelist. They cannot be written by the blueprint loader. Yet every catalog needs them, otherwise facets don't render.

### 20.1 What the blueprint MUST do

- **Nothing inside the blueprint JSON itself.** No flag goes on `SchemaItem` (the `SchemaItem` type has a fixed set of system flags: `isPrice`, `isSku`, `isCurrency`, `isTaxRate`, `isLogin`, `isPassword`, `isSignUp`, `isVisible`, `isProductPreview`, `isCompress`, `isIcon`, `isRatingValue`, `isNotificationEmail`/`Phone*`, `splitPrice`/`splitParts`/`splitUnit` — verified against the `SchemaItem` type definition). There is NO `isFilter` / `isIndexed` / `isFacet` flag — indexing happens automatically via the `'index-data'` Bull consumer for every attribute.
- **In `mapped.yaml` (sidecar for the orchestrator):** emit `post_import_filters[]` task list (see `filters-setup.md` §3) and warning `out-of-whitelist-needs-post-import: filters …` so the orchestrator picks them up.

### 20.2 What the blueprint MUST NOT do

- ❌ Don't write rows to `filters` / `filter_items_mn` / `filter_custom_items_mn` — these tables are not in `ALLOWED_TABLES`; loader throws `BadRequestException("Table 'X' is not whitelisted")` (HTTP 400) at `blueprint-loader.service.ts:111`.
- ❌ Don't set `isFilter` / `isIndexed` / `isFacet` on `SchemaItem` — these keys are NOT in the `SchemaItem` type and no code reads them. They survive inside the `attributes_sets.schema` jsonb (loader doesn't filter unknown nested keys inside jsonb), but contribute nothing — dead weight.
- ❌ Don't add unknown TOP-LEVEL columns to whitelisted-table rows — the loader builds INSERT from raw keys without a column whitelist (`blueprint-loader.service.ts:516`); unknown columns → Postgres rejects → HTTP 500. Validator S27 statically catches this.
- ❌ Don't create `pages` of `general_type_id=4` for "filter dictionaries" — catalogs already use facets via `BaseFiltersService` on the runtime side.
- ❌ Don't hallucinate filter `localizeInfos` title — see §18 (anti-hallucination). Pass `""` (empty string), not `null` — the latter fails `IsLocalizeInfos` validation.

### 20.3 What the orchestrator MUST do

Step 7 in `post-import-orchestration.md`:
1. Resolve the parent `attribute_set` ID per task via `GET /api/admin/attributes-sets` (CRUD list) + client-side filter by `identifier`. (`GET /attributes-sets/marker/:marker` exists but throws `MethodNotAllowedException` — do NOT use.)
2. `POST /api/admin/filters` with body `{ identifier, localizeInfos, scopeTypes }` (camelCase!).
3. `POST /api/admin/filters/:id/items` batch with `{ items: [{ objectType, objectId, attributeIdentifier? }, ...] }` — `objectType='attribute'`, `objectId` = attribute_set ID, `attributeIdentifier` = schema key.
4. Idempotent via `GET /filters/marker-validation/:identifier`.

### 20.4 Source of truth

Universal filter rules — `agents_datasets/rules/filters-setup.md` (real DTO contract, anti-patterns, archetype templates, grep patterns for inspector; runtime model and automatic indexing pipeline are summarised inline in §1 of that file).

## 21. Block attachment — no orphans (S15 strengthening)

A block exists in the OneEntry data model **only to be rendered somewhere**. A block row with no entries in `block_pages_mn`, `block_products_mn`, OR `product_blocks_mn` is **dead code** — it loads into the DB but never appears on the storefront.

### 21.1 Required attachment per block kind

| Block kind (from inspector) | Required attachment | Table |
|---|---|---|
| `static_content`, `carousel`, `hero`, `banner` | one or more pages | `block_pages_mn` |
| `products_collection`, `trending`, `new_arrivals`, `recently_viewed`, `for_you`, `recommendations` | one or more pages | `block_pages_mn` |
| `similar`, `related`, `cross_sell`, `bought_together`, `frequently_ordered`, `wishlist_similar` | one or more **product pages** (or all products of a type) | `block_pages_mn` (page_id of the product page) **or** `product_blocks_mn` (specific product) |
| `reviews`, `faq` | one or more product pages OR a generic page | `block_pages_mn` or `product_blocks_mn` |
| `special_offers`, `sale_promo` | home + sale page | `block_pages_mn` |

### 21.2 What to do when the inspector has the block but no clear scope

1. Use the heuristic suggester (see `post-import-orchestration.md` Step 8 — `suggest_scope_for_block()`).
2. **DO NOT auto-attach** — log a manual task instead. Auto-attachment risks rendering content where the admin didn't expect it.
3. Validator S15 (orphan blocks) emits WARNING, not ERROR — the import proceeds but the report lists each orphan.

### 21.3 Source of truth

Block kinds — `agents_datasets/rules/block-types.md`. Mapper algorithm — `entity-mapper.md` Step 9. Manual-attachment heuristic — `post-import-orchestration.md` Step 8.
